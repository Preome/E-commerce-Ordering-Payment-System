1. IMPLEMENTATION APPROACH AND RATIONALE

1.1 Why Django + DRF?

Django was chosen as the backend framework for the following reasons:

- Rapid development: Django's "batteries-included" philosophy provides ORM, admin panel, authentication, and CSRF protection out of the box.
- Django REST Framework (DRF): Provides serializers, viewsets, authentication backends, throttling, pagination, and browsable API, all needed for a clean REST API design.
- Migration system: Django's built-in migration framework satisfies the requirement to use database migrations, with full version control of schema changes.
- Ecosystem: Mature packages like drf-yasg (Swagger), django-cors-headers, django-redis, and django-environ integrate seamlessly.

1.2 Why Strategy Pattern for Payments?

The requirements explicitly called for a Strategy pattern to support multiple payment providers (Stripe, bKash) without modifying core order logic.

Implementation:
- PaymentProvider (ABC): Defines 5 abstract methods: create_payment, confirm_payment, verify_payment, process_webhook, get_provider_name.
- StripePaymentProvider: Implements Stripe PaymentIntent flow with webhook signature verification.
- BKashPaymentProvider: Implements bKash Tokenized Checkout with token management, checkout, execute, and query flows.
- PaymentProcessor: Static registry context that dispatches calls to the appropriate provider.

Rationale: This allows adding future providers (PayPal, SSLCommerz, etc.) by creating a single new class and registering it in apps.py ready(). No changes to Order, Checkout, or any other code are needed.

1.3 Why DFS for Category Hierarchy?

The requirements specified DFS (Depth-First Search) for product recommendation via category hierarchy.

Implementation:
- Category model uses a self-referential ForeignKey (parent) to create a tree.
- CategoryHierarchy.dfs_traverse() recursively walks from root to leaf nodes.
- get_related_products() collects ancestor and descendant categories, then finds products in those categories.
- Results are cached in Redis (1-hour TTL) to minimize DB calls.

Rationale: DFS naturally fits tree traversal for hierarchical categories. Combined with Redis caching, repeated traversals are served from cache.

1.4 Why SQLite in Docker instead of PostgreSQL?

The docker-compose.yml was simplified to use SQLite instead of PostgreSQL because:
- The .env file contains a Neon PostgreSQL URL for production use.
- Docker falls back to SQLite when DATABASE_URL is not set, making local Docker development zero-config.
- The Dockerfile installs libpq-dev for psycopg2, so PostgreSQL support is ready when DATABASE_URL is provided.

1.5 Why Token Authentication?

DRF TokenAuthentication was chosen over JWT for simplicity:
- No additional library needed (built into DRF).
- Token is created on login, sent in Authorization header.
- Sufficient for the scope of this project.
- SessionAuthentication also enabled for Swagger UI and admin access.

1.6 Why UUID Primary Keys?

All models use UUIDField as primary key instead of auto-incrementing integers:
- Globally unique IDs prevent enumeration attacks.
- No collision risk in distributed systems.
- Standard practice for REST APIs.


2. REJECTED ALTERNATIVES

2.1 Node.js / Express.js (Rejected)

The assessment allowed Node.js/Express.js as an alternative. Rejected because:
- Django provides built-in ORM, migrations, admin, and auth, reducing boilerplate significantly.
- DRF's serializer system is more structured than Express middleware chains.
- Django's migration system is more mature than Sequelize/Knex alternatives.
- The requirement for OOP classes (User, Product, Order, Payment) maps naturally to Django models.

2.2 FastAPI (Rejected)

FastAPI was considered but rejected because:
- No built-in ORM (requires SQLAlchemy or Tortoise ORM setup).
- No built-in migration system (requires Alembic configuration).
- No built-in admin panel.
- Less mature ecosystem for payment integrations.
- Async-first design adds complexity for synchronous payment flows.

2.3 JWT Authentication (Rejected)

JWT was considered over Token Authentication but rejected because:
- Requires additional library (djangorestframework-simplejwt or PyJWT).
- Token refresh/blacklist logic adds complexity.
- DRF TokenAuthentication is simpler and sufficient for this scope.
- PyJWT is included for potential future JWT needs but not actively used for auth.

2.4 Factory Boy for Testing (Rejected)

Factory Boy was considered for test data generation but rejected because:
- Django's TestCase with setUp() and create() calls is straightforward.
- Factory Boy adds another dependency and learning curve.
- The current test suite is manageable without it.

2.5 Celery for Async Tasks (Rejected)

Celery was considered for async payment processing but rejected because:
- Webhook handlers process payments synchronously (sufficient for this scale).
- No background job queue requirements in the spec.
- Could be added later for payment polling, email notifications, etc.

2.6 PostgreSQL in Docker (Rejected for simplicity)

A PostgreSQL service was originally in docker-compose.yml but removed because:
- Adding a database container required more configuration and startup time.
- SQLite works for demonstration and local development.
- The system falls back to PostgreSQL when DATABASE_URL is set (production).

Two levels of testing were implemented:

A) Unit Tests (per app):

B) API Integration Tests (tests/ directory):
   - Test complete request/response cycles through DRF's APIClient.

5. DESIGN PATTERNS AND ALGORITHMS

5.1 Strategy Pattern (Payment System)

PaymentProvider (ABC) defines 5 abstract methods: create_payment, confirm_payment, verify_payment, process_webhook, get_provider_name.

PaymentProcessor (Context/Registry) provides: register_provider, get_provider, create_payment, confirm_payment, verify_payment, process_webhook, available_providers.

Concrete implementations: StripePaymentProvider and BKashPaymentProvider.

Registered in payments/apps.py ready() at Django app startup. Stripe is the default provider.

Adding a new provider:
1. Create PayPalPaymentProvider(PaymentProvider) with 5 methods
2. Register in apps.py: PaymentProcessor.register_provider('paypal', instance)
3. Add 'paypal' to Payment.PROVIDER_CHOICES
4. No changes to Order or Checkout logic needed

5.2 DFS Category Traversal Algorithm

CategoryHierarchy.dfs_traverse(root_categories) recursively walks the category tree:
- Takes a list of root categories
- For each category, appends {category, depth, product_count} to results
- Recurses into children with depth + 1
- Returns the complete traversal list

Used for:
- GET /api/v1/products/categories/hierarchy/
- Product recommendations (ancestor + descendant traversal)

5.3 Redis Caching Strategy

Cache Key                     TTL       Invalidation
---------------------------   -------   ----------------------
category_tree                 1 hour    On category CRUD
category_hierarchy_dfs        1 hour    On category CRUD
recommendations_{product_id}  30 min    TTL expiry only

Falls back to LocMemCache when Redis is unavailable.

5.4 Stock Management Algorithm

On Order Creation:
- Validate all products are active and in stock
- Calculate subtotals: subtotal = price * quantity (in OrderItem.save())
- Calculate total: total = sum of all subtotals

On Payment Success:
- Reduce stock: product.stock -= order_item.quantity
- Validate: raises ValueError if insufficient or negative

On Paid Order Cancellation:
- Restore stock: product.stock += order_item.quantity
- Mark order as canceled
